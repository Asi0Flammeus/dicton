Executive verdict

Same map, different first move.

Opus and Codex converge on the core pattern: many modules are adapters in shape but not in responsibility. The strongest shared diagnosis is that Dicton has several hypothetical seams: one adapter, one implementation, no real behavioral variability, and low Locality.

Where Opus is stronger

It names the callback ouroboros.

Opus is sharper on the `SessionService` → `DictationController` → `SessionService` callback loop. My proposal saw the session split, but Opus exposes the exact construction smell in `container.py`.

Where Codex is sharper

It isolates the output bug.

My highest-priority candidate — active text target replacement — is more concrete. Opus folds it into Candidate 1, but does not foreground that `replace_selection` is currently passed and then ignored.

Main caution

Do not over-delete adapters.

Opus sometimes reads “one adapter” as “delete it”. One adapter means a hypothetical seam, yes — but the response can be to deepen, collapse, or keep as a future-proof test surface. Deletion is not automatically the right move.

Agreement

The shared diagnosis is strong.

The two proposals agree on the structural shape of the problem. They differ mainly in surgical order.

Shared pattern

Extraction has produced files and protocol-looking seams, but ownership often stayed where it was. That creates shallow modules: interfaces that callers must understand almost as deeply as the implementation.

  • `TextProcessorAdapter` and `TextOutputAdapter` wrap bound methods of the caller.
  • `AudioCaptureAdapter` and `STTAdapter` both wrap the same recognizer and split chunk lifecycle.
  • `prompts.py` bypasses injected provider state and redoes fallback.
  • `state_machine.py` names a state machine but does not own transitions.

Shared end state

Both proposals want fewer hypothetical seams and deeper modules:

  • one owner for dictation session policy,
  • one owner for settings schema/defaults/persistence,
  • one owner for LLM text actions,
  • one pure owner for hotkey state transitions,
  • one coherent recording/transcription pipeline.
Divergence

The disagreement is about sequencing.

Opus prioritizes architectural cleanup around core/orchestration. Codex prioritizes a behavior-shaped seam that likely hides an actual defect.

My main disagreement: Opus should not start by asking “collapse controller or push policy into controller?” before fixing the output interface lie. The `replace_selection` flag is already an architectural promise. It should either be honored or removed. That is the smallest high-signal slice.
Opus bias

Architectural purity first.

Opus wants to settle what “core” means, then reorganize surrounding modules. That is intellectually clean, but it risks turning the first step into a broad design debate.

Codex bias

Bug-shaped vertical slice first.

Codex starts from a concrete behavioral contradiction: `replace_selection` exists but is ignored. That produces a smaller first refactor and an immediate regression test.

What Opus adds

Two points should be imported into the plan.

Opus catches issues that my first proposal either underweighted or missed.

1. `core/processing_mode.py` reads `os.environ`.

This violates the intended dependency rule. The fix is small: pass `enable_advanced_modes` from `AppConfig`, or move enablement policy out of core. This is worth doing early because it makes architecture tests sharper.

2. The controller/service callback loop is the real smell.

My session candidate described responsibility interleaving. Opus names the mechanism: `SessionService` calls `DictationController`, which calls back into `SessionService` through adapters. That is the cleanest evidence that the seam is hypothetical.

Synthesized plan

The merged priority order I would execute.

This order preserves Opus’s architecture insight while keeping the first slice small, testable, and user-visible.

Fix/deepen active text output.
Honor `replace_selection` or remove it. Create a regression test around `ACT_ON_TEXT` replacing selected text. This gives immediate leverage and validates the output seam.
Remove `os.environ` from core.
Small symbolic cleanup. Thread `AppConfig.enable_advanced_modes` into mode enablement or move policy to orchestration.
Resolve `SessionService` ↔ `DictationController` ownership.
Use Opus’s design fork: collapse the controller into the session module, or push policy fully into the controller. Do not keep the callback ouroboros.
Make `TextActions` real.
Make prompts pure and run actions through an injected provider/fallback object. This is a contained, high-confidence seam repair.
Make `AppConfig` the single source of truth.
Unify defaults, env parsing, UI field metadata, persistence, and invalidation. Delete legacy config shims only after tests stop depending on them.
Split `SpeechRecognizer` / recording pipeline.
Separate microphone capture, STT dispatch, chunk lifecycle, and transcription filtering. This is important but more runtime-sensitive.
Make `HotkeyStateMachine` real.
Extract pure transition logic from evdev/pyudev/select plumbing. Do when touching FN behavior or after session ownership is clearer.
Clean up `adapters/config/` naming as fallout.
Correct, but not worth a standalone churn PR unless configuration or metrics modules are already moving.
Final read

How close is Opus?

Close enough that the two proposals should be merged, not debated. Opus has the better architectural naming for the controller/session/config issues. Codex has the better first slice because it starts from a concrete behavior contradiction.

Verdict

Opus proposal quality: high. It is aligned with my findings and adds two useful candidates. Its main weakness is breadth: the top recommendation may trigger a larger design conversation before Dicton gets a concrete seam improvement.

Recommended next action

Write a tiny refactor issue for active text output, and append two Opus-derived guardrails: no env reads in core, and no new bound-method pass-through adapters unless the seam has a named reason to exist.
