03 · The bug Codex caught

The real finding: selected text is never replaced.

Opus framed the controller↔service callback chain as architectural theatre. Codex looked at the same chain and noticed something Opus missed: the replace_selection flag travels all the way from the port interface to SessionService.output_result, and is then silently dropped. ACT_ON_TEXT mode notifies the user "Text Replaced" — but the underlying insert_text call inserts at the cursor instead of replacing the selection.

trace · the dropped flag
# core/ports.py:64-68 — interface promises replacement
class TextOutput(Protocol):
    def output(self, text: str, mode, replace_selection: bool) -> None: ...

# adapters/config/text_processing.py:27 — flag is passed through
def output(self, text, mode, replace_selection: bool) -> None:
    self._output_fn(text, mode, replace_selection)   ✓ forwarded

# orchestration/session_service.py:267-281 — flag is dropped
def output_result(self, text, mode, replace_selection: bool) -> None:
    self._text_output.insert_text(text, delay_ms=50)     ✗ replace_selection unused
    if mode == ProcessingMode.ACT_ON_TEXT:
        self._notifications.notify("✓ Text Replaced", text[:100])  ✗ lies to the user
Codex's #1 isn't an architectural opinion. It's a behaviour bug surfaced through architectural review — exactly what shallow seams are supposed to hide, and exactly what the deletion test is supposed to expose.

This single finding shifts the priority order. Codex's recommendation to start with the output module is correct: it's the only candidate where deepening the seam is also fixing user-visible behaviour, not just improving testability.

04 · What Opus added

Two findings Codex didn't surface.

opus only · #5

Core reads os.environ directly

core/processing_mode.py:104 calls os.getenv("ENABLE_ADVANCED_MODES", ...). The refactor plan's dependency table forbids exactly this — and the violation propagates: is_mode_enabled is called from SessionService.start_recording and SessionService.process_text, so a core function reaches around the app_config injection that orchestration carefully threads through.

Small change, large symbolic value: an import os ban in core/ becomes enforceable. Catches future regressions automatically.

opus only · #7

adapters/config/ is a junk drawer

Five files, none about configuration: latency.py (metrics), metrics.py (port adapter), text_processing.py (port adapters), update_checker.py (ops), config_env.py (settings loader).

Naming-only, low priority. Skip unless candidate D (config) lands — that re-homing happens for free as a side effect.

Opus also went one level deeper on candidate D, naming the four shared/*.py compat shims by file and counting the ~5 test imports that block their deletion.

05 · Where they agree

Five candidates, two paths, same conclusion.

Five candidates appear in both reports with materially the same diagnosis. The convergence is striking because the two passes were independent — same vocabulary (Ousterhout, deletion test, seams), same suspect modules, same root cause.

  • Recording / STT pipeline · both flag SpeechRecognizer as a god-object whose two-adapter split (AudioCaptureAdapter + STTAdapter) couples chunk lifecycle across an artificial seam.
  • Session / controller · both flag the controller-vs-service split as fake separation. Opus is more specific about the mechanism (bound-method adapters); Codex is more specific about the symptom (state observer reaches into _state).
  • Configuration · both flag the four-way duplication (AppConfig, config_env, shared/config, config_logic._DEFAULTS + CONFIG_FIELD_MAP).
  • FN handler · both flag the same trap: state_machine.py is 26 lines of enum, the real transition logic stayed in the 813-line handler.
  • LLM dispatch · both flag the broken seam: SessionService receives a provider, then prompts.py resolves its own provider chain via the factory cache.
When two independent passes converge on the same five seams, the seams are real. The remaining question is sequencing, not detection.
06 · Framing

Same findings, different postures.

codex

Bug-driven, prescriptive

Codex prescribes a single direction for each candidate ("deepen the X module"). The recommended starting point is bug-driven: pick the candidate where the shallow seam is most likely hiding incorrect behaviour, fix that first, ship something visible.

Strength: actionable. Weakness: doesn't surface the posture choice behind a refactor (e.g. for the controller/service split, "deepen one module" doesn't say which one).

opus

Architecture-driven, optionful

Opus presents two postures for the controller/service question ("collapse" vs "push policy into the controller") and asks the user to pick. Recommends starting with the conceptual question — what does "core" mean here? — because it shapes the downstream candidates.

Strength: forces the user to own the design call. Weakness: optional postures can stall the work if the user wants prescription.

Both framings have their place. Codex is the right voice when shipping the next PR. Opus is the right voice when the question is "what shape do we want this codebase to have in six months?" — the architecture-refactor-plan question.

07 · Verdict

Synthesised recommendation.

Stop treating these as competing proposals. They are the same audit with different emphases. The right move is to merge:

  1. Ship Codex's #1 first (the replace_selection drop). It's a behaviour bug, the blast radius is one mode, and the test asserts user-visible behaviour through the port interface — exactly the kind of test the deepened seam unlocks.
  2. Then take Opus's #5 (move os.getenv out of core/). One-line change, enables an import os ban as an architecture test in CI.
  3. Then take the joint #4 / Opus #2 (config consolidation). High leverage, broad migration, but the path is now clear: AppConfig wins, shared/config.py dies, the four shims with five test imports get deleted in one PR.
  4. Then take the joint #6 / Opus #4 (LLM dispatch). Localised, contained blast radius, makes prompts.py testable without SDK.
  5. Then the joint #3 / Opus #1 (controller / session). This is where Opus's "two postures" framing matters — pick (a) collapse or (b) push policy in, then execute.
  6. Then the joint #2 / Opus #6 (recording pipeline). Latency-critical, do it once tests around it are stronger.
  7. Last, the joint #5 / Opus #3 (FN handler). Highest concrete pain, lowest reversibility — do it after the smaller wins build confidence.
Codex caught the bug. Opus caught the dependency-rule violation. The codebase is in better shape with both reports than with either alone.

One observation worth keeping

Both audits, independently, surfaced the same meta-pattern: extraction was performed at the file level but not at the responsibility level. state_machine.py is the canonical example, but capture_adapter.py, stt_adapter.py, text_processing.py, and the four shared/* shims all share the shape — a file moved, a name created, a responsibility unchanged.

A second pass through every refactor PR asking "which module owns this decision now?" — rather than "which module contains this code now?" — would catch most of these in code review.
