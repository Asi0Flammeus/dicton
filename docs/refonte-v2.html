dicton, version maigre.

Un outil opinionated. Une seule mission : transcrire et nettoyer la voix française en moins de deux secondes, de la fin du dictat jusqu'au texte collé dans l'app active. Une étude de cas en programmation efficace : faire une seule chose avec une exactitude obsessionnelle.

État courant : ~7 000 LOC Cible : ~940 LOC runtime Pipeline ≤ 300 LOC Une clé · Groq
01 · Le constat

Une cathédrale hexagonale construite pour un cabanon.

Le dicton actuel comporte 87 fichiers Python répartis sur cinq couches architecturales : core/, adapters/, orchestration/, interfaces/, shared/. C'est le pattern hexagonal classique — ports, adapters, factories, container DI, state-machine. Ce pattern existe pour absorber l'incertitude sur les fournisseurs externes.

Or aujourd'hui, l'incertitude a disparu. Groq gagne sur la latence STT et sur la latence LLM, avec le même endpoint et la même clé API. Le reste n'est plus du choix, c'est du bagage.

Quand la cathédrale est construite pour absorber un futur qui ne vient pas, chaque couche devient un impôt sur la lecture, sur le test, sur la modification.

03 · Le killer feature

Transcrire pendant qu'on parle. Nettoyer une seule fois à la fin.

La technique qui rend le <2 s tenable, même sur des audios longs. Existe déjà dans chunk_manager.py — à préserver, pas réinventer.

L'audio est découpé en chunks pendant le recording, sur les frontières de silence. Chaque chunk part chez Groq dès qu'il est prêt, en parallèle du recording. Quand on relâche, il ne reste qu'à transcrire la queue et faire UN seul appel cleanup sur le transcript joint.

  • Latence end-to-end quasi-constante par durée d'audio. 30s = ~1.15s. 60s = ~1.2s. 4s = ~0.7s.
  • UN seul appel cleanup, peu importe le nombre de chunks. Contexte global = nettoyage cohérent.
  • Pre-chauffage des sockets dès le début du recording.

Audio court (~4 s) · un seul chunk 

t=0.00   HOTKEY DOWN
         ├─ visualizer pygame s'ouvre (waveform live)
         ├─ chunker reçoit les frames audio
         ├─ prewarm socket api.groq.com ─► TCP + TLS + HTTP/2 SETTINGS
         │   (un seul handshake pour STT et cleanup — même host)
         │
         │ ... 4 secondes d'audio en RAM, aucun chunk émis ...
         │
t=4.00   HOTKEY UP — chunker.flush() → 1 chunk (tout l'audio)
         ├─ submit Groq STT (socket warm)        ─[~325ms]─►
t=4.32                                                       transcript brut
         ├─ cleanup gpt-oss-20b (même socket)    ─[~370ms]─►
t=4.69                                                       transcript propre
         └─ paste()                              ─[~30ms]──► texte dans l'app

Latence perçue (relâche → texte collé) : ~725 ms

Audio long (~30 s) · 4 chunks 

t=0.0    HOTKEY DOWN — recording + chunking en parallèle
         prewarm socket api.groq.com, visualizer up
         │
t=0.0 ──── enregistrement audio (chunker observe RMS) ──────────────────────►
         │
t=7.2    silence détecté · chunk #1 (6.0s + overlap) ──► Groq STT ─[700ms]─► texte#1 (t=7.9)
t=14.5   silence détecté · chunk #2 (7.0s)             ──► Groq STT ─[750ms]─► texte#2 (t=15.3)
t=22.1   silence détecté · chunk #3 (7.4s)             ──► Groq STT ─[780ms]─► texte#3 (t=22.9)
         │
         │  ↳ pendant ce temps, visualizer affiche la waveform en continu
         │
t=30.0   HOTKEY UP — chunker.flush() → chunk #4 (queue ~7.5s)
                                                ──► Groq STT ─[780ms]─► texte#4 (t=30.78)

t=30.78  tous les futures résolus (chunks 1-3 finis depuis longtemps)
         joined = " ".join([texte#1, texte#2, texte#3, texte#4])
         cleanup gpt-oss-20b(joined) ─[~370ms · UN SEUL appel]─► (t=31.15)
t=31.15  paste()

Latence perçue (relâche → texte collé) : ~1150 ms

La latence est bornée par STT(dernier_chunk) + cleanup(joint), pas par STT(audio_complet). C'est ça, le mécanisme.

Paramètres de chunking

Valeurs issues du chunk_manager.py actuel. Marchent bien en pratique — embarquées telles quelles comme defaults, configurables via TOML.

Paramètre Valeur Pourquoi
min_chunk_s 6.0 Whisper Turbo dégrade sous 5 s. 6 s = ~15-20 mots FR.
max_chunk_s 20.0 Plafond dur. STT sur 20 s ≈ 800 ms — pire-cas pour la queue.
overlap_s 0.4 Couvre la latence de réaction du détecteur de silence.
silence_threshold -40 dBFS Parole FR ≈ -25/-30, bruit pièce ≈ -50. -40 = séparation propre.
silence_window_s 0.35 Pauses inter-phrase FR = 200-500 ms. 350 ms attrape les pauses sans couper les micro-silences.
04 · Une seule clé Groq

Tout sur Groq. Même host, même socket, une clé.

L'illusion qu'il fallait Gemini ou OpenRouter pour le cleanup s'effondre dès qu'on regarde le catalogue Groq de mai 2026. Groq fait Whisper et des LLMs de qualité, sur le même endpoint api.groq.com, avec la même clé.

La découverte qui simplifie tout

Un seul provider, un seul host, un seul handshake TLS. Les appels STT et cleanup partagent la même connexion HTTP/2 vers api.groq.com. Pas de second prewarm. Pas de hop proxy. Pas de deuxième clé à demander à l'utilisateur.

End-to-end sur audio court : STT 325 ms + cleanup 370 ms + paste 30 ms ≈ 725 ms. Sur audio long (30 s) : ~1.15 s. Marge confortable.

Quatre modèles embarqués, l'utilisateur choisit au wizard

Le wizard de premier lancement teste les quatre modèles cleanup avec une phrase FR-test et affiche pour chacun TTFT mesuré, latence totale, rendu du nettoyage et coût annuel estimé. L'utilisateur pick. Le choix est inscrit dans le TOML (cleanup_model = ...) et modifiable plus tard via dicton config (qui pose juste la question du modèle) ou édition directe du fichier.

  • openai/gpt-oss-20b — Vitesse pure. Cleanup ~370 ms · ~2 €/an à 100 dictations/jour · reasoning natif. Le default proposé.
  • openai/gpt-oss-120b — Équilibre vitesse/qualité. Cleanup ~540 ms · ~4 €/an · reasoning natif, plus de paramètres pour les nuances françaises subtiles.
  • llama-3.3-70b-versatile — Qualité maximale FR. Cleanup ~600 ms · ~9 €/an · top multilingue Meta, meilleur sur idiomes et registres parlés.
  • llama-3.1-8b-instant — Mode économie. Cleanup ~280 ms · <1 €/an · qualité FR juste correcte mais imbattable sur la latence et le coût.
05 · Synthèse

Toutes les décisions sont verrouillées.

  • Provider unique : Groq. Une clé. Même socket STT + cleanup.
  • STT : whisper-large-v3-turbo.
  • Cleanup : 4 modèles embarqués (gpt-oss-20b, gpt-oss-120b, llama-3.3-70b, llama-3.1-8b-instant), choix utilisateur au wizard, swap via dicton config.
  • Chunking parallèle + UN seul cleanup final (§03), paramètres figés au courant.
  • Pre-chauffage HTTP/2 au hotkey-down, un seul host.
  • Hotkey : F2 primary + secondary par défaut, support Fn-key Linux via evdev (~60 LOC).
  • Cross-platform : Linux + macOS + Windows. Hard prerequisite.
  • Install : uv tool install dicton.
  • Update : dicton update = uv tool upgrade + restart daemon.
  • Wizard rich en 5 étapes avec self-test 3 s qui teste les 4 modèles cleanup + dicton config pour re-choisir plus tard.
  • Stats locales JSONL + dicton stats avec temps économisé.
  • Pas de releases GitHub, tags git → CI → PyPI.
  • Visualizer pygame (~150 LOC). Pas de notifications desktop.
  • Suppression hexagonale : ports, adapters, factories, container.
  • Mort du .deb et du PyInstaller.
  • Versioning via hatch-vcs : tag git = version.
  • Main file ≤ 300 LOC. Si dépasse → split (pre-commit hook).

Décisions architecturales clés

Tous les choix défendables séparément. L'ensemble forme un système cohérent.

Orchestrateur
pipeline.py est le SEUL fichier qui possède le cycle de vie. Hotkey (pynput + evdev sur Linux), audio (sounddevice), ThreadPoolExecutor pour les STT chunks, event loop asyncio pour httpx. Cap dur ≤300 LOC, validé par un pre-commit hook.
Installation
uv tool install dicton. Identique sur les 3 OS. Tag git → CI uv publish → PyPI.
Provider unique
Groq seul. stt.py et cleanup.py partagent le même httpx.AsyncClient(http2=True) vers api.groq.com. Un seul prewarm, un seul TLS handshake, un seul pool. Modèle cleanup choisi au wizard parmi 4 embarqués (cf. §04), changeable plus tard via dicton config.
Hotkey
Double binding primary + secondary, défaut F2 sur les deux (configurable via wizard). Support de la touche Fn sur Linux laptops via fn_key.py (evdev, ~60 LOC, import optionnel sur Linux uniquement).
Stack runtime
8 dépendances : sounddevice, pynput, pygame, httpx[http2], platformdirs, typer, rich, evdev (Linux uniquement). Appels REST directs en httpx (pas de groq SDK, pas d'openai SDK). Pas de plyer/desktop-notifier.
Latence
Pre-chauffage api.groq.com au hotkey-down. Chunking pendant le recording. UN seul cleanup à la fin. Audio court ≈ 725 ms · audio long ≈ 1150 ms. Marge ~850 ms sous les 2 s.
Visualizer
pygame en main thread (SDL macOS l'exige), 400×100, waveform live + bar RMS + état "REC" / "processing". Frames via queue.Queue depuis le callback audio. ~140 LOC.
Wizard
rich en 5 étapes : system check · clé Groq avec ping live · hotkey chooser · self-test 3 s qui mesure les 4 modèles cleanup avec breakdown latence + coût/an comparés · toggle autostart. Clé en TOML, chmod 600.
CLI
5 commandes via typer : dicton (run daemon) · dicton wizard (relance setup complet) · dicton config (pose juste la question du modèle cleanup et l'écrit dans le TOML) · dicton stats · dicton update.
Cross-platform
Hard prerequisite. sys.platform inline dans output.py et platform.py. CI matrix 3 OS × 2 versions Python (6 cells) doit passer vert pour merger. Wayland documenté (ydotool) — installer script l'installe au besoin.
Tests
6 tests : silence cut, max cut, overlap, pipeline mocké, recovery chunk-failed, stats summary math. Mocks httpx via respx.
