Recherche de design · dicton

Glossaire utilisateur local & intelligent

Comment apprendre, une fois, qu'alysis s'écrit avec un y — sans jamais avoir à le redire.

2026-05-14 auteur · kaïros pour · asi0 status · pré-décision

00Contexte

Le problème, en une dictée

dicton est un pipeline de dictée sub-seconde en français : capture audio → Groq Whisper (whisper-large-v3-turbo) → cleanup LLM (Groq Chat) → injection clavier. Aujourd'hui, aucun mécanisme ne permet à l'utilisateur d'enseigner au système ses propres termes : noms propres (alysis, asi0, centaur-infra), jargon métier, abréviations idiosyncrasiques.

Quand Whisper transcrit alisis au lieu d'alysis, l'utilisateur doit corriger à la main — à chaque dictée. La friction est légère mais cumulative, et elle érode la promesse « tape à la vitesse de la pensée ».

/*

Corriger une fois. Que la faute ne réapparaisse plus. Que le système se souvienne — localement, sans cloud, sans abonnement, sans rituel.

*/

— cahier des charges, asi0

Points d'insertion identifiés

  • Pre-STT : initial_prompt de Whisper (non exploité — stt.py:30-56)
  • Pendant cleanup : prompt système Groq Chat (cleanup.py:22, config.py:28-43)
  • Post-cleanup : interception du texte retourné avant clipboard (pipeline.py:254-261)
  • Stockage : section TOML dans ~/.config/dicton/config.toml (config.py:17-18)

01Quatre approches

Quatre mécanismes orthogonaux — pas quatre variations du même

Approche 01 · KISS

Substitution déterministe post-cleanup

Un dictionnaire plat, regex, à la toute fin du pipeline.

Mécanisme
Section [glossary] en TOML, dict {bad: good}. Boucle de re.sub(r'\balisis\b', 'alysis', text, flags=IGNORECASE) avec préservation de casse, appliquée juste avant injection clavier.
Stockage
~/.config/dicton/config.toml · section [glossary] · format clé-valeur.
Input user
CLI explicite : dicton correct alisis alysis. Append-only, persisté immédiatement.
Touche
config.py (+section), pipeline.py:254 (+5 LOC), cli.py (+commande). ~40 LOC total.
Latence
0 ms perceptible.
  • Déterministe, débogable, prévisible.
  • Zéro dépendance externe, zéro coût API supplémentaire.
  • Implémentable en une après-midi, testable trivialement.
  • Non-contextuel : si l'utilisateur dicte un texte sur la Grèce antique mentionnant « Alisis » comme nom de personnage, il sera écrasé.
  • Pas de variantes morphologiques (alisis / alisi / alysien…) sans entrées multiples.
  • Friction d'apprentissage : l'utilisateur doit penser à lancer la commande.

Approche 02 · À la source

Whisper initial_prompt biasing

Apprendre au modèle de reconnaissance, pas réparer après.

Mécanisme
Section [vocabulary] = liste de termes. Injectée dans initial_prompt de l'API Groq Whisper à chaque chunk : « Vocabulaire courant : alysis, asi0, centaur-infra… ». Le STT biaise sa décodage vers ces tokens.
Stockage
TOML [vocabulary] · liste plate de mots-clés (pas de paires bad→good).
Input user
dicton vocab add alysis. Optionnel : auto-collecte des mots les plus fréquents dans stats.jsonl.
Touche
stt.py:30-56 (+param initial_prompt), config.py (+section), pipeline.py (passage du prompt). ~30 LOC.
Latence
Négligeable côté serveur Groq ; pas d'overhead local.
  • Corrige à la racine : le STT lui-même reconnaît mieux le terme, plutôt que de patcher l'erreur en aval.
  • Aide aussi les variantes morphologiques (le modèle entend mieux le radical).
  • Aucune correction post-hoc : pas de faux positifs sur du texte qui ressemble mais n'est pas le bon mot.
  • Biais probabiliste, pas garantie : Whisper peut quand même se tromper.
  • Budget tokens limité dans le prompt (~224 tokens chez OpenAI ; comportement Groq à vérifier).
  • Pas de mapping bad→good : ne résout pas le cas où l'utilisateur veut un mot rare au lieu d'un mot fréquent homophone.

Approche 03 · Contextuel

Glossaire injecté dans le prompt cleanup LLM

Confier la substitution au modèle qui nettoie déjà — il sait quand ne pas remplacer.

Mécanisme
Section [glossary] avec entrées enrichies : {bad, good, note}. À chaque appel cleanup, ajout au prompt système d'un bloc : « Termes spécifiques utilisateur : remplace systématiquement “alisis” par “alysis” (nom de marque). N'applique pas la substitution si le contexte indique clairement un autre sens. »
Stockage
TOML [[glossary]] en table-of-tables (entrées riches avec note de contexte).
Input user
dicton correct alisis alysis --note "nom de marque". La note de contexte est optionnelle mais améliore le jugement du LLM.
Touche
cleanup.py:22 (templating prompt), config.py (+structure riche). ~50 LOC ; attention au cap 500 LOC de pipeline.py.
Latence
+50–200 tokens prompt par appel ; coût Groq marginal.
  • Contextuel : le LLM peut épargner les faux positifs (le « Alisis » grec antique survit).
  • Gère les variantes morphologiques sans entrée explicite.
  • Cohérent avec l'architecture existante (cleanup est déjà LLM-based).
  • Non-déterministe : le LLM peut rater une substitution (rare mais possible).
  • Si le glossaire grossit, la fenêtre de contexte se sature et le coût grimpe linéairement.
  • Surface d'attaque prompt-injection élargie (à mitiger comme dans le commit 6a42c2e).

Approche 04 · Zéro friction

Auto-apprentissage par détection de re-dictée

L'utilisateur corrige naturellement ; dicton observe et propose.

Mécanisme
Dicton enregistre dans stats.jsonl chaque dictée (déjà partiellement le cas). Quand deux dictées consécutives dans une courte fenêtre (<30 s) ont une distance Levenshtein faible mais un mot diffère, c'est probablement une correction. À la fin de session (ou via un hotkey), un mini-prompt CLI propose : « Tu sembles avoir corrigé alisis → alysis. L'ajouter au glossaire ? [o/N] ».
Stockage
Détection en mémoire ; ajout in fine dans [glossary] (réutilise un des trois mécanismes ci-dessus pour l'application).
Input user
Aucun input explicite. Confirmation en mode lazy (résumé de session) ou eager (notification après chaque détection).
Touche
stats.py (+champ contenu), nouveau learner.py (~80 LOC), cli.py (+commande review). ~130 LOC, plus complexe à tester.
Latence
0 sur le chemin chaud ; analyse offline.
  • Friction nulle : l'utilisateur fait ce qu'il ferait de toute façon (corriger à la main).
  • Capture les corrections qu'on oublierait d'enregistrer manuellement.
  • Données déjà disponibles dans le pipeline ; bonne synergie avec stats.jsonl.
  • Heuristique imparfaite : faux positifs (l'utilisateur a juste changé d'avis sur la phrase, pas corrigé un mot).
  • Nécessite logging du contenu transcrit — implications vie privée à clarifier (opt-in obligatoire).
  • Ne s'applique pas seule : couche d'apprentissage qui doit déléguer l'application à 01, 02, ou 03.

03Recommandation

Une composition, pas un choix unique

Les quatre approches ne sont pas mutuellement exclusives. Elles opèrent à des étages différents du pipeline et se renforcent. Plan d'implémentation incrémental, du plus rentable au plus ambitieux :

  1. Phase 1 — Approche 01 (KISS). Livre la valeur en une après-midi. Établit le format [glossary] en TOML et la commande dicton correct. Couvre 80% du cas alisis → alysis.
  2. Phase 2 — Approche 02 (initial_prompt). Ajoute le biais à la source avec un coût marginal (~30 LOC). Les deux mécanismes se complètent : Whisper essaie d'éviter l'erreur, le regex la rattrape.
  3. Phase 3 — Approche 04 (auto-apprentissage), opt-in. Réduit la friction d'ajout au glossaire ; s'appuie sur les phases 1 et 2 pour appliquer ce qu'elle apprend.
  4. Phase 4 (optionnelle) — Approche 03. À considérer seulement si les phases 1+2 montrent des faux positifs contextuels gênants en pratique. Sinon, ajouter de la complexité LLM pour résoudre un problème qui n'existe pas serait un anti-pattern.

Décision à valider avec asi0 avant tout code : (a) le mapping [glossary] doit-il être case-sensitive ou insensitive ? (b) la commande de correction prend-elle un argument (CLI) ou un raccourci clavier global (capture le presse-papier) ? (c) le contenu des dictées peut-il être loggé pour la phase 4, ou hash uniquement ?
